### Phase 0 decisions

- No numpy available (apt-get blocked, no root). Wrote `test_grids/gen_no_numpy.py`
  using Python `random.Random(seed=42)` (Mersenne Twister). The grids differ from
  what numpy PCG64 would produce, but that is fine — the oracle is the reference
  binary, not a specific set of cell values.

- Saved reference binary as `spawn_sim_ref` to keep it separate from the optimised
  binary (`spawn_sim` produced by `build.sh`).

- Expected outputs for 512: generated by running reference sequentially.
- Expected output for 2048 random_low: generated as the reference timed run
  (the process was launched before build.sh overwrote spawn_sim; Linux kept the
  old inode in memory, so the correct reference ran to completion).
- Reference determinism confirmed: two independent 2-gen runs on 2048 produce
  byte-identical output (cmp passes).

- 8192 and 32768 input grids NOT generated in Phase 0 per plan instructions.
  (8192 inputs were generated for Phase 1 readiness; expected outputs deferred.)

## 2026-05-25 — Phase 1: Bitplane representation and transpose

### Implementation

`src/transpose.cpp` — scalar `bytes_to_bitplanes` and `bitplanes_to_bytes`:
- Bit layout: for column `c`, word index = `c / 64`, bit position = `c % 64` (LSB = col 0 within each 64-wide group). Little-endian bit order chosen to match standard `>>` / `<<` intuition and align with Phase 2 adder work.
- Process 64 columns at a time, building each uint64_t word completely before storing.
- Zero-initialize via `Bitplane::zero()` before writing.

`src/grid.h` — added `Bitplane::zero()` and `#include <cstring>`.

`tests/test_transpose.cpp` — loads grid file, bytes→bitplanes→bytes, `memcmp` vs original.

8192 memory breakdown: 64 MiB input + 16 MiB bitplane pair + 64 MiB output = 144 MiB (measured 147 MiB including executable overhead). Well within limits.
